Conceitos Fundamentais

2. Conceitos Fundamentais

2.1. Modelo Entidade-Relacionamento (MER)

O Modelo Conceitual é a primeira etapa do projeto de um banco de dados e representa a visão de alto nível do negócio, focando exclusivamente no "o quê" precisa ser armazenado, ignorando completamente o "como" isso será armazenado no software.

Nesta fase, não nos importamos se usaremos MySQL, PostgreSQL, Oracle ou MongoDB; nosso foco absoluto é compreender a realidade do cliente e traduzi-la em uma estrutura gráfica e semântica que faça sentido tanto para os programadores quanto para os gestores da empresa.

2. Conceitos Fundamentais

Linguagem Padronizada

Este modelo utiliza uma linguagem padronizada, geralmente a abordagem Entidade-Relacionamento (ER), que nos permite mapear o cenário de forma abstrata.

A grande vantagem do modelo conceitual é servir como um contrato e um canal de comunicação claríssimo entre a equipe técnica e os usuários finais, garantindo que nenhum dado vital para a operação da empresa seja esquecido antes de começarmos a escrever qualquer linha de código.

2. Conceitos Fundamentais

A Origem do MER

O Modelo Entidade-Relacionamento (MER), proposto originalmente por Peter Chen em 1976, é uma técnica de modelagem conceitual de alto nível amplamente utilizada para descrever a estrutura de um banco de dados de forma abstrata e independente de tecnologias específicas de implementação ou SGBDs.

Ele funciona como uma ferramenta de comunicação entre técnicos e usuários, permitindo capturar os requisitos de dados da organização através de uma percepção do mundo real baseada em objetos e suas associações

2. Conceitos Fundamentais

Notação Estrutural em Texto

Como ainda não estamos no código SQL, nossa demonstração prática do Modelo Conceitual será feita através de uma notação estrutural em texto (que antecede o desenho do diagrama):

# Mapeamento Conceitual - Sistema de Biblioteca [CENÁRIO DO NEGÓCIO] - A biblioteca possui livros que são emprestados para usuários. - Precisamos saber quem pegou qual livro e quando. [ESTRUTURA EXTRAÍDA] Entidade 1: USUARIO Entidade 2: LIVRO Ação: EMPRÉSTIMO (conecta Usuário e Livro)
2. Conceitos Fundamentais

Explicação linha por linha:

  • [CENÁRIO DO NEGÓCIO]: Representa a frase dita pelo cliente na entrevista de requisitos.
  • [ESTRUTURA EXTRAÍDA]: É a tradução da nossa mente de desenvolvedor isolando os substantivos e verbos.
  • Entidade 1 e 2: São as coisas reais que identificamos que precisam ter dados guardados.
  • Ação: É o evento que faz essas duas coisas interagirem no mundo real.

2. Conceitos Fundamentais

2.2. Entidades

Uma Entidade, dentro do contexto de banco de dados e modelagem, é qualquer objeto, pessoa, evento ou conceito do mundo real que possua existência independente e sobre o qual desejamos guardar informações.

Se você olhar ao seu redor, "Aluno", "Computador", "Turma" e "Professor" são entidades clássicas em um ambiente escolar.

Na modelagem conceitual, as entidades são representadas graficamente por retângulos contendo o nome da entidade sempre no singular e em letras maiúsculas.

2. Conceitos Fundamentais

Entidades Fortes e Fracas

É de extrema importância diferenciar uma entidade forte (que existe por si só, como "Cliente") de uma entidade fraca (que depende de outra para existir, como "Dependente" que só existe no sistema se o "Funcionário" pai existir).

Identificar corretamente as entidades é o primeiro passo de qualquer arquiteto de software.

As entidades são representadas por retângulos:

Entidade

Representação estruturada de como definimos uma entidade antes de desenhá-la no software de modelagem:

ENTIDADE: PRODUTO Definição: Representa os itens físicos que a loja vende. Tipo: Entidade Forte (independente). ENTIDADE: HISTORICO_PRECO Definição: Representa as variações de valor de um produto ao longo do tempo. Tipo: Entidade Fraca (só existe se o PRODUTO existir).

  • Tipo: Classificamos para saber se ela precisará herdar identificadores de outra entidade no futuro.

2. Conceitos Fundamentais

2.3. Atributos

Os atributos são as propriedades, características ou qualidades que descrevem uma entidade.

Se a entidade fosse um substantivo ("Carro"), os atributos seriam os adjetivos e especificações ("Cor", "Placa", "Modelo", "Ano").

Todo atributo deve possuir um nome claro e ser vinculado diretamente a uma entidade específica, sendo representados graficamente no diagrama por elipses (círculos ovais) ligadas ao retângulo da entidade.

2. Conceitos Fundamentais

Classificação dos Atributos (Parte 1)

Os atributos podem ser classificados de diversas maneiras, como podemos ver abaixo:

  • Atributo Identificador ou Chave (que é único para cada registro, como o CPF de uma pessoa ou o Chassi de um carro);
  • Simples: São atributos considerados atômicos ou indivisíveis. Por exemplo, o atributo “cor” de uma entidade “carro” é indivisível;
  • Composto: Já os atributos compostos podem ser divididos em partes menores, com significados independentes. Um caso clássico é o atributo “Endereço”, o qual pode ser dividido em “Rua”, “Número”, “Bairro”, “Cidade”, entre outros itens;
2. Conceitos Fundamentais

Classificação dos Atributos (Parte 2)

  • Monovalorado: Esse atributo possui apenas um valor para uma determinada entidade. Por exemplo, na entidade “Pessoa”, cada indivíduo pode ter apenas um valor de CPF;
  • Multivalorado: Já o multivalorado permite que um atributo de uma mesma entidade possa ter mais de um valor. Um exemplo é o atributo “Telefone”, uma vez que uma pessoa pode ter mais de um número de telefone;
  • Armazenado: São aqueles previamente definidos e armazenados em um banco de dados (Data de nascimento).
2. Conceitos Fundamentais

Classificação dos Atributos (Parte 3)

  • Derivado: O derivado é aquele que não é armazenado, mas que pode ser obtido através daqueles atributos armazenados. Por exemplo, suponha-se que o atributo “Data de Nascimento” seja definido em um banco de dados. Com isso, o atributo “idade” pode ser obtido através desse atributo previamente armazenado.
  • Descritivo: Os atributos descritivos retratam as características intrínsecas de uma determinada entidade, como a idade, cor, nome, entre outros.
2. Conceitos Fundamentais

Classificação dos Atributos (Parte 4)

  • Nominativo: Por sua vez, os nominativos, além de serem descritivos, possuem o objetivo de especificar e definir uma entidade. Por exemplo, código, CPF, nome, entre outros.
  • Relacional: São os atributos presentes nos relacionamentos. Um exemplo, são os atributos “Local” e “Data” do relacionamento “Casamento”.

2. Conceitos Fundamentais

2.4. Relacionamentos e Cardinalidade

Os relacionamentos são as associações lógicas e estruturais entre duas ou mais entidades do nosso modelo.

No mundo real, as coisas não existem isoladas; elas interagem. Um "Cliente" COMPRA um "Produto". Um "Professor" MINISTRA uma "Disciplina".

O relacionamento é representado graficamente por um losango (diamante) contendo um verbo que liga os retângulos das entidades.

Os relacionamentos são representados por losangos: Relacionamento

2. Conceitos Fundamentais

Cardinalidade

Junto aos relacionamentos, temos o conceito fundamental de Cardinalidade, que define numericamente como essas entidades se associam e pode ser:

  • 1:1 um-para-um: um gerente gerencia um único departamento;
  • 1:N um-para-muitos: uma mãe pode ter vários filhos, mas o filho só tem uma mãe biológica ou;
  • N:M muitos-para-muitos: um aluno cursa várias disciplinas, e uma disciplina possui vários alunos.

A cardinalidade correta dita como as tabelas do sistema se comportarão no futuro.

2. Conceitos Fundamentais

Entendendo o Diagrama

Para facilitar o entendimento, observe o diagrama abaixo: Mas o que significa aqueles números na figura?

Bom, eles correspondem às cardinalidades das entidades. A cardinalidade da entidade está do lado oposto ao seu retângulo, havendo a cardinalidade mínima e a máxima.

Para lê-las, adotamos a seguinte estrutura:

ENTIDADE + RELACIONAMENTO + CARDINALIDADE + ENTIDADE

2. Conceitos Fundamentais

Leitura da Cardinalidade

Desse modo, as cardinalidades de cada uma das duas entidades são: Um professor pode ministrar 0 (mínimo) ou ‘n’ (máximo) disciplinas.

Uma disciplina pode ser ministrada por apenas 1 (mínimo) ou 1 (máximo) professor.

O ‘n’ significa mais de uma, ou seja, um professor pode ministrar nenhuma disciplina ou mais de uma.

Já quando as cardinalidades mínima e máxima são iguais, é porque ela está restrita a apenas aquele número.

Assim, uma disciplina pode ser ministrada apenas por um professor.

2. Conceitos Fundamentais

Leitura Semântica

Estruturando a leitura semântica de um relacionamento e sua cardinalidade:

RELACIONAMENTO: COMPRA Entidade A: CLIENTE Entidade B: CARRO Regra de Negócio: - 1 Cliente pode comprar (N) Carros. - 1 Carro é comprado por apenas (1) Cliente. Cardinalidade Final: 1:N (Um para Muitos)
2. Conceitos Fundamentais

Explicação linha por linha:

  • RELACIONAMENTO: COMPRA: Identificamos o verbo da ação.
  • Entidade A e B: As pontas da conexão.
  • Regra de Negócio: Lemos a relação nos dois sentidos. Do cliente para o carro, e do carro para o cliente.
  • Cardinalidade Final: A união das regras define que este é um relacionamento 1:N, o que significa que futuramente a chave primária do cliente vai migrar como chave estrangeira para o carro.

2.5. Regras Básicas de Modelagem

A modelagem de dados é um processo fundamental que exige rigor metodológico e padronização para traduzir requisitos informais de negócio em estruturas precisas e eficientes.

Essa prática não se resume à criação de diagramas visuais, mas sim à construção de um esquema formal e não ambíguo que serve como base para a implementação do banco de dados.

Abaixo, os conceitos fundamentais das regras básicas de modelagem são detalhados:

2. Conceitos Fundamentais

Clareza na Nomenclatura: O Uso do Singular

A primeira regra estabelece que as entidades devem ser nomeadas com substantivos no singular (ex: "CLIENTE" em vez de "CLIENTES"). Essa convenção é crucial por dois motivos conceituais:

  • Classe vs. Coleção: Uma entidade representa a intenção da relação (o molde ou definição estrutural), enquanto o conjunto de dados preenchidos representa a extensão ou o estado da relação. Nomear no singular reforça que estamos definindo o tipo de objeto e suas propriedades, e não listando seus registros.
  • Significado Semântico: Nomes curtos e familiares ao negócio facilitam a comunicação entre técnicos e usuários finais, tornando o modelo auto documentado.
2. Conceitos Fundamentais

Identificação Obrigatória: A Chave Primária (PK)

Toda entidade forte deve obrigatoriamente possuir um atributo identificador, conhecido no modelo relacional como Chave Primária (PK).

  • Integridade de Entidade: A função primordial da chave primária é garantir a identidade única de cada ocorrência (tupla), assegurando que nenhum componente da chave seja nulo e que não existam duplicatas. Sem um identificador bem definido, torna-se impossível distinguir ou recuperar um objeto específico do mundo real dentro do sistema.
2. Conceitos Fundamentais

Identificação Obrigatória: A Chave Primária (PK) (Continuação)

  • Natureza do Identificador: O identificador deve ser mínimo, contendo apenas o conjunto essencial de atributos necessários para a distinção única. No nível conceitual, a identificação e a descrição são construtos distintos: o papel da chave é identificar a existência da entidade, não descrevê-la.
2. Conceitos Fundamentais

Evitar a Redundância Precoce

Na fase de modelagem conceitual, a regra de ouro é "um fato em um lugar", visando minimizar a redundância para prevenir anomalias de atualização.

  • Minimalismo de Dados: O modelo deve seguir a regra do dado mínimo: "Tudo o que é necessário está lá e tudo o que está lá é necessário". Repetir atributos em entidades diferentes sem necessidade de negócio gera inconsistência, pois uma alteração em um local pode não ser refletida no outro, gerando dados conflitantes conhecido como princípio GIGO: Garbage In, Garbage Out.
2. Conceitos Fundamentais

O Papel do Relacionamento

No Modelo Entidade-Relacionamento (MER), as associações entre entidades são representadas por relacionamentos, e não pela repetição manual de campos.

A conexão física entre as informações (como o uso de Chaves Estrangeiras) é um detalhe de implementação que será resolvido na fase de projeto lógico/físico.

Se uma informação pode ser derivada através de um relacionamento existente, armazená-la novamente tornaria o atributo redundante e o modelo ineficiente.

2. Conceitos Fundamentais

Comparação de Mapeamentos

Comparação de um mapeamento fora do padrão versus um mapeamento profissional:

# ERRADO (Fora do Padrão)

Entidades: Vendas, Usuarios

Atributos Vendas: id, quem_comprou, data_da_venda, cpf_do_usuario

Relacionamento: Faz

# CORRETO (Padrão de Mercado)

Entidade: VENDA

Atributos: codigo_venda (Identificador), data_venda

Entidade: USUARIO

Atributos: cpf (Identificador), nome

Relacionamento: REALIZA (Associa Usuario 1:N Venda)

2. Conceitos Fundamentais

Explicação linha por linha:

No modelo incorreto, usamos plural ("Vendas"), misturamos dados que não pertencem à entidade ("cpf_do_usuario" dentro de venda) e usamos verbos fracos ("Faz").

No modelo correto, usamos singular ("VENDA", "USUARIO"), separamos as responsabilidades, garantimos um identificador para cada e deixamos o verbo formal ("REALIZA") unir as informações.

3. Casos Reais do Mercado

Caso 1: Aplicativos de Delivery (iFood / UberEats)

A modelagem conceitual dessas empresas é gigantesca, mas o coração da plataforma gira em torno do relacionamento entre entidades principais: RESTAURANTE, CLIENTE, ENTREGADOR e PEDIDO.

O pedido é uma entidade central que se relaciona com as outras três simultaneamente.

O endereço de entrega não é um atributo simples do cliente, pois um cliente pode ter vários endereços (Casa, Trabalho), logo "ENDEREÇO" se torna uma entidade própria, relacionada ao cliente por uma cardinalidade 1:N.

3. Casos Reais do Mercado

Caso 2: Plataformas de Streaming (Netflix)

Para garantir o funcionamento do sistema de recomendações, a modelagem distingue perfeitamente a entidade CONTA (quem paga, possui o cartão de crédito) da entidade PERFIL (quem assiste: você, seu irmão, seção Kids).

A relação é de 1 Conta para N Perfis. O histórico de visualização, chamado ASSISTIU, é um relacionamento N:M complexo entre PERFIL e CONTEÚDO, que inclusive guarda atributos no próprio relacionamento (como os minutos parados do filme ou a nota dada).

FAQ – Dúvidas Reais de Alunos

5. FAQ

Dúvidas Reais de Alunos (Parte 1)

Explicação

1. Qual a diferença entre modelo conceitual, lógico e físico?

O conceitual foca nas regras de negócio e é focado no ser humano (diagrama). O lógico traduz isso para a tecnologia de tabelas e chaves (focado em como o software vê a estrutura). O físico é o código SQL real rodando no SGBD (PostgreSQL, MySQL).

Explicação

2. Um atributo pode virar uma entidade?

Sim! Isso acontece com frequência. Se um atributo como "Endereço" começa a precisar de muitos sub-dados (rua, cep, cidade, estado, país) e a empresa precisa gerenciar rotas de entrega baseadas nessas cidades, "Endereço" deixa de ser um atributo composto e vira sua própria entidade "ENDERECO".

5. FAQ

Dúvidas Reais de Alunos (Parte 2)

Explicação

3. Como eu sei se um relacionamento é 1:N ou N:M?

Fazendo a pergunta nos dois sentidos do verbo. "Um autor escreve quantos livros? Vários (N). Um livro pode ser escrito por quantos autores? Vários (M)". Se deu 'vários' dos dois lados, a relação é N:M.

Explicação

4. Preciso colocar todos os dados do mundo em uma entidade?

Não! Você só mapeia o que é relevante para o sistema. Num sistema de locadora de carros, saber o "Tipo Sanguíneo" do cliente é inútil. Em um sistema de Hospital, é um atributo vital. O cenário dita os atributos.

5. O que é um atributo identificador?

É aquele dado que NUNCA se repete entre duas entidades do mesmo tipo. Pessoas podem ter o mesmo nome, mesma data de nascimento e morar na mesma rua. Mas não podem ter o mesmo CPF. O CPF é um identificador forte.

Roteiro do Projeto

O que praticamos hoje não é apenas um exercício acadêmico. Lembra do Roteiro do Projeto? Um item obrigatório da sua documentação técnica e entrega final para a banca é exatamente o Modelo Entidade-Relacionamento (MER), diagrama conceitual.

O levantamento que vocês fizeram na Aula 4 se transformou no diagrama de hoje. As entidades, atributos e relacionamentos que vocês definirem agora para o sistema de vocês serão a espinha dorsal de todo o código SQL que programaremos a partir das próximas aulas.

Se o modelo estiver errado, a aplicação inteira vai quebrar na hora da programação.
